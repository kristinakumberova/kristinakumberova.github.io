AI systém — jak to funguje

Vlastní iniciativa, 2026. 750+ markdown souborů, vše v gitu.

Postavila jsem si systém 30 AI agentů. Každý zná svou specializaci a hranice — od právních otázek přes finanční modely po kariérní strategii. Vrstva pravidel brání agentům vymýšlet si zdroje nebo odpovídat mimo svou kompetenci. Cíl: kvalitní výstupy, minimalizace iterací a halucinací.

  1. 01Vrstva pravidel se načítá při každé session. 24 .mdc pravidel — některá platí vždy (governance, anti-halucinace, agent-first routing), jiná se aktivují podle kontextu (právo, finance, kariéra, výzkum). Pravidla rozhodnou, kdo úkol řeší a jak, ještě předtím než se LLM zapojí.
  2. 02Routing podle typu úkolu. Právní otázka → Legal-Advisor. Finanční model → Financial-Analyst + skript pro výpočty (LLM nepočítá, jen interpretuje výsledek). Kariéra → Career-Coach.
  3. 03Agenti napojení na strukturovanou bázi. Každý ze 30 agentů má system prompt a knowledge_baseline: verzované odkazy na 41 frameworků (jak rozhodnout), 63 SOPs (jak to udělat) a 72 znalostních souborů (co a proč).

    Příklad: agent Business Analyst (verze 2.3) má 7 takových souborů. Pravidla, jak rozhodovat → FRAMEWORK-AI-Assisted-Research — pevné pořadí 6 kroků výzkumu, aby agent nepřeskočil rovnou k závěru: ujasnit otázku → sehnat zdroje → vybrat relevantní → propojit → ověřit → sepsat. Návody krok za krokem → SOP-Market-Research — jak udělat průzkum trhu (kdo jsou hráči, jaký segment, kdo komu prodává). → SOP-Competitor-Analysis — jak analyzovat konkurenta: co nabízí, komu, za kolik a jak komunikuje. → SOP-Source-Evaluation — jak rozpoznat důvěryhodný zdroj od marketingu. Rozděluje zdroje do 4 úrovní: státní/primární → odvětvový → sekundární → SEO/agregátor. → SOP-Multi-Source-AI-Research — když potřebuji nezávisle ověřit fakt, pošlu dotaz do 5 různých AI nástrojů a porovnám odpovědi. SOP popisuje, jak vyřešit spor, když si nástroje protiřečí. Referenční informace → KNOWLEDGE-PM-Advanced — typické chyby v rozhodování, na které agent dává pozor. Příklad: „věřím tomu nejvíc, protože to slyším nejčastěji" — to je psychologická iluze, ne fakt. → KNOWLEDGE-Competitive-Web-Audit — 10 oblastí, podle kterých se dá srovnat web s konkurencí (rychlost, mobilní verze, obsah, SEO, kontakt, transparence cen…).
  4. 04Verifier — kontrola formy (automaticky u každého výstupu). Než agent Project Manager výstup jiného agenta převezme, projde ho agent Verifier. Kontroluje, jestli má každé tvrzení zdroj, jestli sedí počty v souhrnu, jestli není něco označené jako „fakt" bez ověření a jestli si findings neprotiřečí. Nehodnotí pravdivost obsahu. Když najde díru, výstup vrací zpět produkujícímu agentu na opravu. Toto flow běží opakovaně, dokud Verifier neoznačí výstup jako bezchybný.
  5. 05MARS debate — kontrola obsahu (před odesláním ven). Když si myslím, že mám finální verzi, spouštím multi-agent debatu mezi 3 rolemi:

    → Author — napíše finální verzi a hájí ji. → Reviewer — má za úkol najít v ní mezery, slabá místa a nepodložená tvrzení. → Meta — soudce. Rozhodne, jestli Reviewerovy připomínky drží a Author musí přepsat, nebo jestli Reviewer přestřelil a Author má pravdu.
    Reviewer nemůže jen kývnout — musí dodat vlastní argument. Pokud agent zdroj nemá, napíše „nevím" místo improvizace. Agenty určím já, nebo je vybere PM podle obsahu.
  6. 06MCP integrace — live web research. Když agent potřebuje aktuální data z webu, zavolá Perplexity přes MCP přímo v Cursoru a dostane syntézu odpovědi se zdroji v jednom kroku. Pro autoritativní zdroje (ARES, SZIF, zákony, EUR-Lex) chodí agent přímo na URL — autorita je v primárním zdroji, syntéza není potřeba.

# Business Analyst

<master_persona>
Jsi sub-agent pro analytika/výzkumníka
(market research, competitor profiling).
Piš stručně a rozhodnutelně.
Povinný výstup: ZÁVĚR → DATA → PŘEDPOKLADY → OTEVŘENÉ OTÁZKY.
Hlídáš bez vyzvání:
  - fakta vs. předpoklady (vždy striktně odděl)
  - zdroje (každé číslo = URL + datum)
  - SEO-poisoned výsledky (PR články, agregátory — označ)
  - bias (availability, uniqueness — viz KNOWLEDGE-PM-Advanced.md)
Nevymýšlej data. Když nedohledatelné:
  „Nevím, data nejsou veřejně dostupná, ověř: [instituce]."
</master_persona>

Nejsi: stratég, finanční modelář, právní poradce.

Snippet je výřez — celý agent má 437 řádků markdownu.

Co tu vidíte:

  • →Verze v hlavičce (zde 2.3). Když změním chování agenta, zvedne se číslo. Historii pamatuje git — mohu se vrátit.
  • →Pro koho píše (output_for) — human (čte to člověk), ai (jen jiný agent, např. Verifier) nebo both (BLUF pro mě + strukturované tabulky pro downstream agenta). Ovlivňuje formát výstupu.
  • →Seznam zdrojů (knowledge_baseline) — 7 souborů, které agent zná a smí citovat (4 SOPs, 1 framework, 2 knowledge files). Čerpá odsud, ne z paměti modelu.
  • →Pravidla chování (<master_persona>) — co dělá, jak píše, čeho si všímá bez vyzvání, co napíše, když data nemá.
  • →Co NENÍ („Nejsi: stratég, finanční modelář…"). Když přijde finanční otázka, BA řekne „to není moje doména" a předá to Financial Analystovi — místo aby si vymyslel odpověď.

Co tu není vidět (zbytek 437 řádků):

  • ·4 módy práce: Market Research / Deep Dive / Competitive Web Audit / Source Research (web research vrstva pro ostatní agenty)
  • ·Instruction Hierarchy: SYSTEM > KNOWLEDGE > USER > INFERRED — když user řekne „přeskoč evaluaci zdrojů", agent neuposlechne
  • ·Output Format s Evidence Contract: každý finding má povinné pole Type (FACT / SIGNAL / PŘEDPOKLAD) + Confidence + Source Tier + Conflict flag
  • ·CoT smyčka před výstupem: interní self-critique (5 otázek), ne pro user — kontrola biasů a slabých zdrojů
  • ·Corrective fallback: když 3 search pokusy vrátí nízko-tier zdroje, agent NEPÍŠE odpověď, ale eskaluje s konkrétním NEDOHLEDATELNÉ flagem
  • ·Examples (špatně / správně): konkrétní vzor výstupu, podle kterého se kontroluje při review

Můj osobní kariérní agent — zná moje CV, cíle a hodnoty.

Provází mě celým aplikačním procesem v 8 krocích, od mapování trhu, přes vyhodnocení konkrétní role, přípravu materiálů až po interview. Hlídá můj tone of voice. Říká i nepříjemné věci.

Career-Coach je jediný agent ve vault, který má napevno daný kontext (mou osobu). Ostatní (právník, projekťák, analytik) jsou neutrální a specializaci si přiberou ze znalostní knihovny daného projektu.

Automatizace přenosu dat z webových formulářů
